# Sing Like a Pro — Webinar Landing Page + Admin Panel

A single-page webinar registration funnel (cloned/redesigned from the
`live.nishantchopra.com` reference) with a full admin panel for managing
webinars and site content. Next.js App Router, no CMS — content lives in
your own Postgres database.

## Stack

- **Next.js 16** (App Router) + React 19 + TypeScript
- **Tailwind v4** — custom design tokens in `src/app/globals.css` (see
  comments there for the color/type system)
- **Drizzle ORM** + **Neon** (serverless Postgres, HTTP driver)
- **better-auth** — email/password, single admin account (no public sign-up)
- **react-hook-form + zod** — shadcn-style `Form` primitive
  (`src/components/ui/form.tsx`) wired to Zod validation

## How content flows

- The public page (`/`) always renders whichever row in `webinars` has
  `isActive = true`. There's no per-webinar URL — creating a new "campaign"
  in `/admin/webinars` and marking it Live is how you switch what's shown.
- Testimonials, bonuses, FAQs, feature highlights, and audience segments are
  a **shared content pool** — edited once, shown regardless of which
  webinar is active. If you want any of these to vary per-campaign later,
  it's a one-column migration (add a `webinarId` FK to that table).
- Money is stored as integer paise (₹99 = `9900`).
- Registration is an **external redirect** — each webinar has its own
  `registrationUrl`; there's no payment processing in this app itself.

## Project layout

```
src/
  app/                     routes only — thin, no business logic
    page.tsx               public landing page
    admin/
      login/                public sign-in
      (protected)/          everything behind the session check
        webinars/, testimonials/, bonuses/, features/, audience/, faqs/,
        testimonial-screenshots/, settings/
    api/auth/[...all]/      better-auth route handler
  features/
    webinar/                public-page section components + read queries
    admin/                  one folder per admin resource: schema.ts
                            (zod), queries.ts (all DB access), actions.ts
                            (server actions), *-form.tsx (RHF form),
                            components/ (shared list/form/table pieces)
  db/
    schema.ts              Drizzle schema (content tables)
    auth-schema.ts          generated by better-auth's CLI — don't hand-edit
    index.ts                Neon + Drizzle client
  components/ui/            shadcn-style primitives (button, input, form, …)
```

## Local setup

1. **Create a Neon project** at neon.tech, copy the pooled connection
   string.
2. Copy the env file and fill it in:
   ```bash
   cp .env.example .env.local
   ```
3. Install dependencies (this project was built with npm in the sandbox
   that produced it, but `bun install` works too since there's no
   npm-specific lockfile committed):
   ```bash
   npm install
   ```
4. Push the schema to your database:
   ```bash
   npm run db:push
   ```
5. Seed placeholder content so the page renders something immediately:
   ```bash
   npm run db:seed
   ```
6. Create your admin login (sign-up is disabled in the UI on purpose):
   ```bash
   npm run create-admin -- "admin" admin@gmail.com "Admin@123"
   ```
7. Run the dev server:
   ```bash
   npm run dev
   ```
   Public page: http://localhost:3000 — Admin: http://localhost:3000/admin

## Replacing placeholder content

The seed script fills every table with placeholder copy and
`placehold.co` / `i.pravatar.cc` images so the page isn't empty on first
run. Replace all of it from `/admin` — there's a CRUD screen per content
type. Real testimonial screenshots, mentor photos, and video thumbnails
should go to your own storage (S3, GCS, Cloudinary, etc.); paste the
resulting URL into the relevant admin form. If you add a new external image
host, add it to `remotePatterns` in `next.config.ts`.

Headlines support two inline highlight tags so non-technical edits still
get the two-tone treatment seen on the page:
`In {gold}3 Simple Steps{/gold} Master... {red}Live Masterclass!{/red}`

## Deploying to a Hostinger VPS + Neon

Regular Hostinger **shared hosting can't run this** — Next.js needs a
persistent Node.js process (server actions, dynamic rendering). You need a
**Hostinger VPS** (or Cloud) plan with root/SSH access. Neon stays exactly
as configured locally; nothing changes there.

On the VPS (Ubuntu, matches the Caddy + systemd pattern used for your other
services):

```bash
# 1. Install Node.js (or Bun) and Caddy if not already present
curl -fsSL https://deb.nodesource.com/setup_lts.x | sudo -E bash -
sudo apt-get install -y nodejs caddy

# 2. Clone and build
git clone <your-repo-url> /var/www/nishant-chopra-webinar
cd /var/www/nishant-chopra-webinar
cp .env.example .env.local   # fill in production DATABASE_URL, secrets, URL
npm install
npm run build
npm run db:push
npm run db:seed              # first deploy only
npm run create-admin -- "You" you@example.com "strong-password"

# 3. Run it as a service (systemd example)
sudo tee /etc/systemd/system/nishant-webinar.service > /dev/null <<'UNIT'
[Unit]
Description=Nishant Chopra webinar site
After=network.target

[Service]
WorkingDirectory=/var/www/nishant-chopra-webinar
ExecStart=/usr/bin/npm run start
Restart=always
Environment=NODE_ENV=production
EnvironmentFile=/var/www/nishant-chopra-webinar/.env.local

[Install]
WantedBy=multi-user.target
UNIT

sudo systemctl enable --now nishant-webinar

# 4. Reverse proxy with automatic HTTPS
sudo tee -a /etc/caddy/Caddyfile > /dev/null <<'CADDY'
your-domain.com {
  reverse_proxy localhost:3000
}
CADDY
sudo systemctl restart caddy
```

Redeploying after changes is `git pull && npm install && npm run build &&
sudo systemctl restart nishant-webinar`.

## Known limitations / good next steps

- **No atomic transaction on "Set as Live"** — Neon's HTTP driver doesn't
  support interactive transactions, so switching the active webinar is two
  sequential statements. Fine for a single-operator admin panel; switch to
  `drizzle-orm/neon-serverless` (websocket/Pool) first if that ever matters.
- **No image upload** — admin forms take image _URLs_, not file uploads.
  Wire up S3/GCS/Cloudinary presigned uploads if you want that.
- **No drag-to-reorder** — ordering is a plain number field on each item.
- **Registration is a plain redirect** — no lead capture, no payment
  webhook. If you later want real payment (Cashfree, given your other
  project) or lead capture before redirecting, that's a bigger addition —
  happy to scope it separately.
